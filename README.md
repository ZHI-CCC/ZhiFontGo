# ZhiFontGo

ZhiFontGo 是一个面向 Minecraft 1.21.1 NeoForge 客户端的高质量字体渲染项目。它的核心目标不是简单地“把原版字体换掉”，而是围绕中文服务器、定制化视觉表现、启动阶段预热、缓存复用、世界文本兼容、Iris 光影兼容以及高频文本场景的稳定性，构建一整套可长期维护的客户端字体解决方案。

这个项目诞生的出发点很明确：原版字体在中文环境中的表现往往不能满足定制服、重内容整合包以及强调视觉风格的服务器需求。很多现成方案要么只是替换 UI 字体，要么无法覆盖世界中的文字、玩家名、展示实体、告示牌、提示框与物品标签，要么在加载性能、缓存命中、光影兼容、字形清晰度这些实际体验上存在明显短板。ZhiFontGo 选择直接从运行时渲染链路入手，构建一套同时关注“效果”和“工程可控性”的实现。

从用户体验角度看，ZhiFontGo 提供的并不是一个孤立的字体切换按钮，而是一整条完整链路：在 `resourcepacks/font` 中发现外部字体文件，允许玩家在游戏内选择字体、调整字号、采样、偏移和阴影，再通过缓存系统把常用字形预构建到磁盘，在下一次启动时直接复用，并在进入游戏前就尽量把常用字库准备完成，减少首次遇到新文字时的顿卡。它解决的是一个真正落地的问题，而不是一个只在截图里好看的演示功能。

## 项目定位

ZhiFontGo 主要服务于以下几类场景：

1. 中文服务器或中文整合包，希望统一 UI、聊天、物品名称、世界文字和提示文本的整体视觉风格。
2. 对自定义文本效果有要求的客户端，例如彩色标签、边缘发光、脉冲、渐变、高亮、隐藏控制标签等。
3. 希望在保持字体清晰度的前提下，降低重复字形构建开销、缩短二次启动加载时间的项目。
4. 需要兼顾 Iris 光影、世界空间文字、实体名称牌、展示文本、告示牌与调试类界面的复杂渲染环境。
5. 需要对客户端使用域名做约束，或者希望将字体渲染作为服务器定制体验的一部分进行统一交付的项目。

换句话说，ZhiFontGo 不是一个“字体皮肤包”，而更像是一个围绕文字渲染体验搭建的客户端基础设施层。

## 核心能力

### 1. 运行时字体替换

项目的核心在于运行时替换 Minecraft 的字体渲染入口，而不是只在静态资源层做覆盖。这样做的好处是，项目可以根据当前激活字体、世界文本兼容模式、阴影设置、Iris 状态和自定义效果标签，在真正绘制文本时做出不同决策。

项目通过 mixin 和自定义 `Font` / `FontSet` 访问链路，接管原版字体创建与部分文本渲染路径。这样不仅能覆盖界面文本，还能进一步覆盖世界空间中的文本对象，例如：

- 实体名称牌
- 告示牌文字
- 文本展示实体
- BetterF3、Jade 一类客户端信息组件
- 世界掉落物名称和数量标签

这也是 ZhiFontGo 与普通资源包式替换方案的本质区别之一。

### 2. SDF 字体渲染

项目内部采用 SDF 思路构建字形纹理，而不是单纯输出一张普通位图。SDF 渲染的优势在于：

- 在不同尺寸下拥有更稳定的边缘表现
- 更适合做发光、描边、脉冲、渐变等特效
- 在高采样下能维持较好的清晰度
- 对后续 shader 风格扩展更友好

项目中与此相关的关键组成包括：

- `ZhiFontGlyphProviderManager`
- `ZhiFontAtlasManager`
- `ZhiFontTextureManager`
- `ZhiFontShaderManager`
- `assets/zhifontgo/shaders/core/zhifont_sdf.*`

这一套实现说明项目并不是“把 Java 字体画成图片贴上去”那么简单，而是在纹理生成、图集管理和运行时绘制上都有比较明确的设计。

### 3. 外部字体发现与游戏内选择

ZhiFontGo 支持从客户端资源包目录下的 `resourcepacks/font` 自动发现外部字体文件，支持的常见格式包括：

- `.ttf`
- `.otf`
- `.ttc`

项目会把发现到的字体转成内部条目，供界面选择、预览和应用。对应功能主要分布在：

- `ZhiFontDiscoveryManager`
- `ZhiFontSelectScreen`
- `ZhiFontManager`

玩家不需要手动改配置文件，也不需要每次替换资源包后重启整个环境来测试。对于一个强调字体效果调校的项目来说，这是非常重要的使用门槛优化。

### 4. 预览与参数调校

ZhiFontGo 提供了完整的预览链路，而不是“选了字体直接全局生效”。玩家可以在应用前先预览不同设置组合，包括：

- 字号
- Oversample 采样倍数
- 水平偏移
- 垂直偏移
- 阴影开关

这种设计非常适合处理中文字体在 Minecraft 中常见的“看起来偏高”“笔画发虚”“字框太紧”“阴影太脏”之类问题。不同字体之间差异巨大，如果没有预览与微调，最终效果很难稳定。

## 缓存系统

### 为什么需要缓存

高质量字体渲染最贵的部分，不是把文字画到屏幕上，而是首次构建大量字形时的成本。尤其在中文环境下，常用字符数量远大于英文，若每次启动都从零构建，必然带来明显的等待与顿卡。

因此，ZhiFontGo 内建了一套字形缓存机制。缓存系统不是附属功能，而是项目体验是否成立的关键组成。

### 缓存目录与命名

缓存文件保存在：

`resourcepacks/font/.zhifontgo-cache`

文件名会编码字体名称和关键参数，例如：

`MiSans-Semibold__s8_00_o6_00_x0_00_y0_00__622c486b9d69.bin`

这种命名方式有两个优点：

1. 人眼可读，能直接看出对应的是哪套字体和哪组参数。
2. 通过哈希区分具体字形缓存版本，避免不同字体文件或不同参数之间互相污染。

### 缓存内容

缓存记录的不是一张最终整图，而是字形级别的数据，包括：

- codepoint
- 字形 SDF 像素
- 宽高
- advance
- bearingLeft
- bearingTop
- 是否为空白字形

这样做意味着：

- 可以按需重新组装图集
- 可以对不同渲染路径复用相同字形数据
- 可以在兼容模式和正常模式之间共享前置字形构建结果

### 缓存策略

项目当前缓存链路包含以下几个阶段：

1. 依据字体文件内容与渲染参数生成缓存键。
2. 启动时优先查找磁盘缓存。
3. 命中时优先读取缓存，减少重复计算。
4. 对未命中的常用字形做补齐。
5. 仅在新增或变更时回写缓存。

这个策略的意义在于：ZhiFontGo 不是“有缓存就读，没缓存就整套重做”这么粗糙，而是尝试做到“缓存命中优先，缺口补齐，补完落盘，下一次继续复用”。

## 启动阶段预热

启动阶段预热是 ZhiFontGo 非常关键的一条链路。项目的目标不是把问题推迟到游戏内，而是在用户真正进入主菜单或世界之前，尽量把常用字库准备好。

这样做的收益很直接：

- 避免第一次看到某个界面或文本时突然卡一下
- 减少世界中出现新名字牌、新掉落物标签、新提示文本时的构建延迟
- 让“字体替换”更接近一个已经准备完成的系统，而不是边用边补

项目中这一部分主要由：

- `ZhiFontReloadManager`
- `ZhiFontLoadingManager`
- `ZhiFontManager`
- `ZhiFontLoadRequestData`
- `ZhiFontPreparedFontData`

共同完成。

这里的设计难点不是“能不能预热”，而是：

- 预热时机要足够早
- 不能与其他模组的初始化链路强烈冲突
- 需要尽量利用已有缓存
- 对于缓存缺失的字形，需要在这个阶段补齐
- 预热完成后要能直接安装到运行时字体系统中

这也是项目在持续迭代中最需要谨慎处理的部分之一。

## 世界文本与 Iris 兼容

世界文本兼容是 ZhiFontGo 另一个很有含金量的方向。很多字体替换项目只能处理 GUI，而一旦进入世界空间文本，尤其是与 Iris 光影结合后，就会出现各种问题：

- 字体变回原版
- 文本过糊
- 发光效果不正确
- 玩家名或生物名不可见
- 展示实体文本渲染异常

ZhiFontGo 为此引入了单独的兼容思路：

- 普通 SDF 渲染路径
- Iris 兼容世界文本路径

项目中相关结构包括：

- `ZhiFontIrisCompatManager`
- `ZhiFontAtlasManager.PipelineMode`
- `ZhiFontRenderTypeManager`
- `ZhiFontTextDisplayRendererMixin`
- `ZhiFontEntityRendererMixin`
- `ZhiFontSignRendererMixin`

当检测到 Iris 或需要世界兼容模式时，项目不会简单粗暴地回退全部效果，而是尽量让世界文本走更合适的图集与渲染策略。这说明项目在设计上明确区分了“菜单字体好看”和“世界文字真的能稳定用”这两件事。

## 自定义文本效果

项目支持一套面向文本内容的效果标签体系，用于在服务端或客户端文本中嵌入表现逻辑。根据现有代码和迭代方向，这套能力主要用于：

- 颜色标签
- 渐变
- 发光
- 闪光
- 脉冲
- 隐藏控制标签但保留逻辑信息
- 对聊天、悬浮提示、名称牌等文本进行定制表现

相比直接硬编码颜色值，这种标签化设计的意义在于：文本生产端可以把“内容”和“表现”半分离。对于服务器欢迎语、NPC 标题、公告、标签系统、任务说明、掉落物名称、世界文字，这种方式会非常实用。

## 掉落物名称与数量标签

项目除了字体替换，还扩展到了世界中的掉落物展示。对应的 `ZhiFontItemLabelManager` 负责在世界渲染阶段给掉落物附加更清晰、可定制的名称与数量显示。

这个功能的价值并不仅仅是“好看”，它还会直接影响玩家的信息获取效率。特别是在：

- 大量掉落堆叠
- 自定义物品名
- 需要颜色分级的数量信息
- 稀有掉落突出显示

这些场景里，一个好用的世界标签系统往往比单纯的聊天提示更直观。

## 配置与状态管理

项目的状态管理主要围绕以下几类数据展开：

- 当前选中的字体
- 字号、采样、偏移、阴影设置
- 字体缓存键与缓存文件名
- 启动时是否需要自动恢复上次字体

相关类包括：

- `ZhiFontSettingsData`
- `ZhiFontSettingsManager`
- `ZhiFontData`
- `ZhiFontLoadRequestData`
- `ZhiFontPreparedFontData`

这种拆分的好处在于，项目把“用户偏好”“一次加载请求”“运行时字体实例”“缓存描述”这几类概念分得比较开。对于长期维护来说，这一点非常重要。

## 服务端辅助子项目

仓库中除了主 NeoForge 客户端模组，还包含一个 `spigot` 子项目。这个子项目目前更像一个桥接组件，用来配合客户端做服务端侧的一些辅助工作。

从当前仓库状态看，它已经包含：

- Spigot 插件入口
- 插件消息通道注册
- 与客户端定制体验相关的基础桥接能力

虽然它不是本仓库的渲染主体，但它说明 ZhiFontGo 的设计并不只盯着本地客户端，而是在考虑“客户端模组 + 服务器生态协作”这件事。

## 工程结构概览

`src/main/java/org/zhi/zhifontgo`

- `manager`
  这里是核心实现层，包含字体准备、缓存、预热、图集、纹理、渲染器、世界标签、Iris 兼容、设置、域名限制等关键逻辑。
- `mixin`
  这里负责接入原版与其他常见文本渲染路径，把自定义字体能力真正接到游戏运行时。
- `screen`
  提供字体选择、预览和加载过程界面。
- `data`
  定义字体、加载请求、加载结果、缓存描述、设置数据等结构。
- `listener`
  负责监听客户端事件与渲染阶段事件，把系统挂到 NeoForge 生命周期里。
- `util`
  提供资源定位和字符集构建等基础辅助。

`spigot`

- 一个独立的子工程，用于服务端桥接和配套辅助。

## 构建环境

项目当前基于：

- Java 21
- NeoForge ModDev Gradle 插件
- Minecraft 1.21.1
- NeoForge 21.1.x
- Parchment mappings

主工程使用 `net.neoforged.moddev` 插件，子项目中还包含一个 Spigot 构建链路。整体上，这个仓库更适合已经熟悉现代 NeoForge 开发环境的开发者接手或继续扩展。

## 适合继续扩展的方向

ZhiFontGo 已经具备很强的基础能力，但它也天然适合继续向下列方向发展：

1. 更细粒度的缓存分层与命中统计。
2. 更明确的启动阶段监控与日志输出，帮助定位“缓存命中率”和“补字耗时”。
3. 更丰富的文本效果 DSL，使服务器文本策划可以更方便地编写动态标签。
4. 更稳的 Iris / Sodium / FancyMenu / DrippyLoadingScreen 等复杂客户端环境兼容处理。
5. 更多世界空间文本对象的统一接管。
6. 与服务端桥接更完整的消息协议，例如提及提示、服务端下发标签规则等。

## 为什么这个项目值得认真维护

很多人低估了“文字”在 Minecraft 客户端体验中的地位。实际上，菜单、聊天、名称牌、告示牌、提示框、掉落物、任务文本、服务器欢迎语、说明书、地图说明，几乎所有高密度内容交互都离不开文字。一个能真正把文字系统做深的项目，往往不是边角料，而是服务器定制体验最基础也最常被忽视的一层。

ZhiFontGo 的价值就在这里。它不是追求一个单点特效，而是在尝试把字体渲染、缓存、启动时机、世界空间文本兼容、参数调校、服务端协作这些本来分散的问题连成一套可运行的体系。只要这条路线继续走稳，它就不仅是一个“能换字体”的模组，而会成为中文定制客户端体验里非常核心的一块基础设施。

## 结语

如果你正在寻找一个能够真正服务中文环境、支持高质量字体表现、考虑启动预热与缓存复用、并且愿意正面处理世界文本和光影兼容问题的 NeoForge 项目，那么 ZhiFontGo 就是这样一个项目。

它也许还在持续打磨中，某些链路仍需要继续压测和修正，但它已经具备了一个成熟方向所需要的关键特征：明确的问题意识、足够深入的运行时接入、可扩展的数据结构、对真实使用场景的强关注，以及愿意为“体验是否真的成立”不断迭代的工程态度。
